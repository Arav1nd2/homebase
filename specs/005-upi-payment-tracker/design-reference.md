## Tokens & components in play

These are the general, closed-system constraints every HomeBase surface (including an undesigned one) must build within — pulled because `plan.md`/`tasks.md` reference UI work that will touch each of them:

- **Radius/borders (`DESIGN.md` §Space, shape, depth):** none, ever — "no card, no panel, no rounded rectangle anywhere in this system." Hairline (1px) dividers (`--border-subtle`/`--border`) are the only structural device. `components/upi-tracker/*` (qr-scanner, confirm-prompt, tag-picker, transaction-list, summary-card) must follow this — no card chrome around any of them.
- **Motion (`DESIGN.md` §Motion):** 200–300ms opacity/small-transform only; no bounce/parallax/looping motion; `prefers-reduced-motion` collapses to instant. Applies to the confirm-prompt's appearance and any step-transition in the scan→amount→tag→pay flow.
- **Type/register split (`design/VOICE.md` §Register split):** `--font-display` (Newsreader) at `--text-lg`+ is licensed *only* for first-ever-use empty states, one moment per page, never for a CTA or form field. Everything else — CTAs, form fields, dense-data rows, loading/error/success — uses `--font-body` (Inter) via the existing `type.dense`/`type.cta-label`/etc. aliases.
- **Functional colors (`DESIGN.md` §Functional colors):** a **closed** set — `error`/`success`/`warning`/`info` at fixed hues (25°/145°/85°/240°), muted chroma, independently AA-verified. There is no fifth functional color and no general-purpose categorical/qualitative color ramp. This directly bears on tag colors — see Open Gaps.
- **Redundant-cue rule (`DESIGN.md` §Redundant-cue discharge):** standing rule — because the accent is green and error is red, every state distinction (~10% of male users are red-green colorblind) must carry a non-color cue too (icon glyph, shape, strikethrough, or text), documented per component in a table. No entry exists yet for a transaction-status indicator — see Open Gaps.
- **`ToolCard` organism (`DESIGN.md` §ToolCard, lines 778–801):** the *only* way any tool — including UPI Payment Tracker — gets a Launcher entry, via a **closed props set**: `id`, `label` (canonical name), `mark` (one glyph, display serif), `route`, `toolType` (closed enum: `heatmap`\|`ledger`\|`checklist`\|`photo-card`), `badge` (`{ kind, value, unit? }`, kind ∈ `count`\|`streak`\|`balance`\|`none`), `pinned`. "The 7th-tool proof" (line 797) explicitly shows adding a tool is meant to be *just* supplying these — but two of them have no answer yet for this feature (`mark`, `toolType`) — see Open Gaps.
- **`Dense-frame` (`DESIGN.md` §Dense-frame, lines 803–816):** shared chrome across exactly four documented interiors (heatmap/ledger/checklist/photo-card). The transaction history view (`history/page.tsx`) is a dense list and will want this chrome, but its actual row shape (payee + amount + tag chip(s) + status) matches none of the four interiors exactly — closest is the Ledger interior's three-column row (description / date-payer meta / right-aligned tabular-numeral amount), minus its "you owe/you're owed" balance-line semantics, plus tag chips and a status indicator neither interior has.
- **Primary CTA (`DESIGN.md` §Primary CTA):** one per page/screen-state, bottom-fixed full-width bar, accent fill, no border at rest, Fitts-sized. Applies to each step of the scan/amount/tag/pay flow and to "Confirm & save"-equivalent actions.
- **Form field (`DESIGN.md` §Form field):** top-aligned label, 16px input text (iOS zoom floor), on-blur validation (never per-keystroke), three redundant error cues (boundary + icon + message). **Forgiving amount parsing is explicitly specified** ("12", "12.5", "$12.50", comma decimals all accepted, normalized on blur) — directly relevant to the amount step (FR-003/FR-004) and to `data-model.md`'s `amountPaise` integer storage: normalization to paise should happen client-side on blur, per this pattern, not by rejecting non-canonical input formats.

## Voice & copy

From `design/VOICE.md`:

- **Voice (stable, all four attributes apply everywhere in this feature too):** Measured (no breathless pacing/exclamation-mark defaults), Exact (real numbers/names, e.g. state the actual payee/amount, never "your recent payment"), Companionable-not-chirpy (quiet acknowledgment at real completion, no mascot/confetti), Plain-about-limits (failure named directly, no blame toward the user, the AI, or — for this feature — the UPI app/network).
- **Register split:** literary/verse register is licensed *only* for first-ever-use empty states, "all 6 tools ONLY" (VOICE.md's own wording) — this feature's empty states (e.g., first-ever transaction history) are not currently covered by that license at all; everything else (scan step, amount/tag forms, confirm prompt, error states) is unambiguously Plain/scannable register regardless.
- **Error formula (Yifrah: what → why → fix, no blame, no dead-end)** — directly applicable to this feature's several first-class error states: malformed QR (FR-021), no UPI app installed (FR-020), camera permission denied (FR-022). The worked example (Expenses' AI-parse failure, VOICE.md lines 57–72) is the closest structural precedent: name what happened in plain terms, an external/ordinary cause (never "you scanned it wrong"), two live recovery paths, no dead-end. E.g., a malformed-QR message should follow the same shape: what ("that code isn't a UPI payment link") → why, ordinary and blameless ("codes for other apps or a damaged code can look like this") → fix ("try scanning again, or [enter the payee manually]").

## Open gaps

This feature needs real design-for-ai work before its UI is built — these are not things `/speckit-plan`/`/speckit-tasks` should have silently invented, and weren't:

1. **Canonical name.** `spec.md`'s title "UPI Payment Tracker" violates JOURNEY.md's own naming convention for the other six tools ("short, parallel noun-phrases naming the content domain, not the mechanism — no 'tracker'/'splitter'/'book' suffix survives into the canonical form," line 184). Needs a short noun-phrase form (e.g., something parallel to "Expenses"/"Groceries") chosen the same deliberate way the other six were.
2. **`ToolCard.mark` glyph.** No punctuation mark has been chosen for this tool (Expenses uses "§", Recipes "¶", etc.) — required before it can get a real Launcher entry.
3. **`ToolCard.toolType`.** None of the four closed dense-frame families (`heatmap`/`ledger`/`checklist`/`photo-card`) is a clean fit for a personal transaction log with per-row tag chips and a 4-value status. This is a structural, DNA-level decision (adapt "ledger," or a genuine 5th family) — per this skill's Revision discipline, that's a `/design-for-ai:plan` → `/design-for-ai:mock` → `/design-for-ai:build` pass, not an ad hoc reuse decided inside a Route Handler or component file.
4. **Tag color palette.** Confirmed here (not just flagged speculatively in `research.md` §9): the locked color system has no categorical/qualitative multi-hue set for open-ended user-defined tags — only the closed neutral/accent/functional tiers. A muted-chroma, colorblind-verified qualitative ramp needs to be designed (matching the existing `palette.mjs`-generated, Lab/OKLCH-based method) before `tags/page.tsx`'s color-swatch picker (`tasks.md` T038) can be built correctly.
5. **Transaction status indicator.** The 4-value `pending`/`success`/`failed`/`unconfirmed` status has no existing component analog — every current status-like pattern in `DESIGN.md` is binary (checklist checked/unchecked, heatmap complete/incomplete) or a directional 3-state tied to a numeric balance (ledger). Needs its own redundant-cue design (a words/icon-glyph pairing, added to the §Redundant-cue discharge table) before `confirm-prompt.tsx`/`transaction-list.tsx` are built.
6. **No page spec at all** for any of this feature's screens: the scan step (this is also the *only* HomeBase tool that uses the device camera — no existing pattern to extend), the amount/tagging steps, the confirm prompt, the history/filter/summary view, the manual-backfill form, or tag management.
7. **Empty-state register.** VOICE.md's literary-verse license is explicitly scoped to "all 6 tools ONLY" — a 7th tool's first-use empty state isn't currently licensed for that register at all; extending it needs a documented amendment to VOICE.md, not a silent assumption either way.
